# レビュー チェックリスト

記事生成後、2 段階のレビューを実施する。

- **Step 5a (機械的チェック)**: 本体エージェントがフロントマター / 件数 / URL / 長さ 等を自己チェック
- **Step 5b (独立レビュー)**: サブエージェントが文脈リセット状態で記事を読み、内容の質と整合性を評価

どちらかが Fail なら全体 Fail。

## Step 5a: 機械的チェック

各項目を Pass / Fail で判定する。1 つでも Fail があれば 5a Fail。

| # | 項目 | 判定基準 |
|---|---|---|
| 0 | フロントマター | ファイル先頭に YAML フロントマターがあり、`title` / `date` / `description` / `tags` の 4 キーがすべて存在。`date` は `YYYY-MM-DD` 形式。`description` は 60〜140 文字。`tags` は非空配列 |
| 1 | トピック数 | `## トピック` 配下に `### N. <タイトル>` が 5 件以上 |
| 2 | 出典 URL の存在 | 各トピックに `出典: <URL>` 行が 1 つ以上 |
| 3 | URL の実在性 | 各 URL のドメインが `references/sources.md` の情報源ドメインと一致 (huggingface.co, arxiv.org, news.ycombinator.com, b.hatena.ne.jp, anthropic.com, openai.com, deepmind.google) または当該情報源からリンクされている外部ドメイン |
| 4 | 本文の長さ | 各トピックの要約が 100 文字以上 |
| 5 | エラー文言の不在 | 本文に「情報を取得できませんでした」「APIエラー」「データなし」「Error」「Failed」等の文言を含まない |
| 6 | 重複の不在 | トピックのタイトルが類似 (80% 以上一致) する組がない |
| 7 | カテゴリの妥当性 | 各トピックの `カテゴリ:` が 5 種類 (`論文 / 公式リリース / 議論 / ツール / その他`) のいずれか |
| 8 | ハイライトの存在 | `## 今日のハイライト` 配下に箇条書き 3 件以上 |

## Step 5b: 独立レビュー (サブエージェント)

`Agent` ツール (subagent_type=general-purpose) で起動する。**生成過程は一切共有しない**。プロンプトに記事ファイルパスだけを渡し、レビュアーは初めてその記事に触れる読者として評価する。

### 独立レビュー プロンプト

サブエージェントには以下のテンプレートをそのまま渡す (`{{path}}` は実ファイルパスに置換):

````text
あなたは AI ニュースレターの編集者です。以下の Markdown 記事を
**初めて読む読者の立場**でレビューしてください。記事の生成過程は
共有されていません。記事ファイルだけを Read し、必要に応じて出典 URL を
WebFetch で確認した上で、以下の 5 観点を Pass/Fail で評価します。

対象ファイル: {{path}}

### 評価観点

| # | 観点 | Fail の例 |
|---|---|---|
| A | 出典との整合性 | 出典 URL の主張と本文の説明が食い違う、出典に書かれていない事実を本文が断定している |
| B | 重要度順 | トピック 1 が明らかに地味で、後ろの方に大型リリースが埋もれている |
| C | 情報密度 | トピックが冗長で水増し / 逆に短すぎて中身がない / 5〜10 分で読み切れない分量 |
| D | 可読性 | 専門外の読者が読んで意味が取れない、専門用語が説明なしに連続する |
| E | フロントマター品質 | `description` がハイライトと一致しない / 内容を誤って要約している、`tags` が記事本文のカテゴリと乖離している |

### 出力フォーマット

以下の JSON ブロックを **1 つだけ** 出力してください。前後に説明文を入れない。

```json
{
  "overall": "pass" | "fail",
  "items": [
    {"id": "A", "result": "pass" | "fail", "comment": "<1〜2 文>"},
    {"id": "B", "result": "pass" | "fail", "comment": "<1〜2 文>"},
    {"id": "C", "result": "pass" | "fail", "comment": "<1〜2 文>"},
    {"id": "D", "result": "pass" | "fail", "comment": "<1〜2 文>"},
    {"id": "E", "result": "pass" | "fail", "comment": "<1〜2 文>"}
  ]
}
```

1 つでも `fail` があれば `overall` は `fail`。
````

### レビュアーの WebFetch 利用方針

- 出典 URL の確認は **観点 A の評価に必要な範囲で** 行う (全 URL を fetch する必要はない)
- 主要 3 トピック程度の出典をスポットチェックすれば十分
- fetch 失敗時はその URL は判定対象から除外する (Fail にしない)

## 統合判定フロー

1. Step 5a を実行 → Pass / Fail を記録
2. Step 5b を実行 → サブエージェントの JSON を解釈し、`overall` を記録
3. 5a と 5b の **両方 Pass** なら全体 Pass、PR 作成へ
4. どちらかでも Fail → 全体 Fail、再生成へ

## 再生成時の指示

Fail だった項目を明示し、その項目を満たすように再生成する。情報源の取得をやり直すかどうかは Fail 内容による:

- **5a 由来**
  - トピック数不足 (項目 1) → 情報源を再取得し、より多くのエントリを拾う
  - フロントマター (項目 0) → 既存記事に対しフロントマター部分のみ修正
  - URL/エラー文言問題 (項目 3, 5) → 取得済みデータの整形をやり直す
  - 本文長さ・カテゴリ・重複 (項目 4, 6, 7) → 既存データの再整理のみ
- **5b 由来**
  - 観点 A (出典との整合性) Fail → 情報源を再取得し、出典との突き合わせをやり直す
  - 観点 B (重要度順) Fail → トピックの並べ替えのみ
  - 観点 C (情報密度) Fail → 各トピックの要約を書き直す
  - 観点 D (可読性) Fail → 専門用語の補足や言い回し改善
  - 観点 E (フロントマター品質) Fail → `description` / `tags` を本文に合わせて再生成

## 失敗時の最終動作

2 回目の生成でも Fail の場合:

- 記事ファイル `news/YYYY-MM-DD.md` はそのまま (生成された状態で)
- PR は draft で作成
- PR に `needs-review` ラベルを付与
- PR 本文に Fail した項目とその理由を記載 (5a / 5b の結果両方)
